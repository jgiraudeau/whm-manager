# Project Memory

Last update: April 4, 2026

## Current Situation
- The inter-account WordPress migration flow is completely dependent on our custom Fallback strategy because o2switch definitively blocked Softaculous Backup/Restore for inter-account usage (`off_backup_restore`).
- We have completely overhauled the Fallback file copy process migrating from a slow file-by-file transfer (40 minutes) to a **fast ZIP stream architecture (Phase 2)**.

## What Was Done (April 3-4, 2026)
- **ZIP Migration Architecture**:
  - We abandoned the file-by-file copy for a bulk compression approach using `cpanelApi2` `Fileman::fileop` `compress` operation.
  - **Identified cPanel Limitations**: `Fileman::fileop compress` silently fails if `sourcefiles` contains directories formatted with nested paths.
  - **Final Solution Implemented**: We now compress the *entire* WordPress directory from its parent (e.g. zipping `wp1` from `public_html`).
- **Streaming Pipeline for Huge Files**:
  - Because zipping an entire WP installation includes heavy cache/backup folders (leading to 1GB+ files), we can no longer load the file completely in RAM.
  - Developed a **Stream-based download/upload model** in `cross-account-wordpress-fallback.ts`:
    - The ZIP is generated by cPanel on the source server.
    - We use Node.js `stream/promises pipeline` to download (`GET /viewer/...`) and write the ZIP directly to local instance ephemeral disk (avoiding RAM crashes).
    - We then upload the local chunked file via `FormData` streaming to the target cPanel server using `Fileman::upload_files`.
    - Once extracted on target, the temporary ZIP is properly cleaned up from all ends.
